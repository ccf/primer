# User Journeys

This document keeps the roadmap anchored to how Primer is meant to be used in practice.

## 1. Engineer Improvement Journey

Goal: help an individual engineer get more effective with AI-assisted work.

1. Open **My Profile** to see effectiveness, friction, and recommendations.
2. Open **Growth** to review workflow playbooks and learning paths.
3. Open **Sessions** to inspect a representative successful or failed session.
4. Turn a recommendation into an **Intervention** when there is a concrete change to try.
5. Use the **MCP sidecar** during future sessions to reinforce the pattern in workflow.

Success signal:
- higher effectiveness score
- lower repeated friction
- better quality outcomes
- reduced cost per successful outcome

## 2. Team Lead Coaching Journey

Goal: identify repeatable high-value patterns and remove team bottlenecks.

1. Start in **Organization** to spot who or what needs attention.
2. Open **Growth** to see patterns, cohort differences, and playbooks worth spreading.
3. Open a **Project** workspace to understand where readiness or workflow friction is blocking the team.
4. Create a project- or team-scoped **Intervention**.
5. Return later to compare before/after metrics and decide whether to standardize the change.

Success signal:
- better adoption of successful workflows
- fewer repeated bottlenecks on key projects
- improved team-level effectiveness and quality

## 3. Platform / Admin Enablement Journey

Goal: improve the environment around engineers, not just individual behavior.

1. Start in **Projects** to see which repos are hard to use effectively with AI.
2. Review project enablement recommendations tied to real bottlenecks.
3. Inspect linked repositories, readiness signals, and workflow hotspots.
4. Launch an **Intervention** for repo docs, toolchain fixes, permissions, or guidance.
5. Measure whether the project became easier, safer, and cheaper to work in.

Success signal:
- higher repo readiness
- lower friction in repeated workflows
- more consistent outcomes across projects

## 4. Interactive Tour Outline

Primer should eventually ship a short guided tour that follows the product loop:

1. **Profile**
   Show personal effectiveness, friction, and "what this product is for."
2. **Growth**
   Explain workflow playbooks and how peer patterns become guidance.
3. **Project Workspace**
   Show how repo readiness and bottlenecks shape outcomes.
4. **Interventions**
   Show how recommendations turn into tracked actions.
5. **Sessions**
   Show the evidence layer behind every recommendation.
6. **Sidecar**
   Close with how Primer should support engineers during work, not only after it.

## 5. How To Use This With The Roadmap

Before shipping a new roadmap item, answer:

- Which user does this primarily help?
- Which page or in-workflow moment should expose it?
- What decision becomes easier because of it?
- What metric should move if it works?
- How does it connect to the rest of the loop: insight -> action -> measurement?

If a feature cannot be placed in one of these journeys, it likely needs a clearer product role before implementation.
